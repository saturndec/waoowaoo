# 生成图片与重生参数错误修复说明

## 背景与问题分析
- 现象：在「点击开始创作」或图片生成流程中，前端出现“生成失败，请求参数不正确，请检查后重试”。
- 现象补充：重生分组（`regenerate_group`）也会报“请求参数不正确”，后端日志为 `missing server-generated billingInfo for billable task type: regenerate_group`。
- 现象补充：单张重生（`regenerate-single-image`）同样存在该链路风险。
- 根因一：`generate-image`、`regenerate-group`、`regenerate-single-image` 路由在项目未配置 `characterModel` / `locationModel` 时，没有在提交任务前显式失败，导致 billable 任务进入 `submitTask` 后被统一判定为参数错误或 billingInfo 缺失。
- 根因二：`getProjectModelConfig` 注释声明“项目配置 > 用户偏好 > null”，但原实现仅 `analysisModel` 有用户偏好回退，图片相关模型无回退，导致“用户已配置默认模型但项目读取为 null”的场景。
- 根因三：图像 capability 必填项（如 `resolution`）缺失时，接口直接回传内部错误串（`CAPABILITY_REQUIRED: ...`），用户无法判断应在哪配置。
- 根因四：`gemini-compatible` 的 `baseUrl` 被配置成 endpoint（如 `/v1/images/generations`）时，SDK 会继续拼接 `/v1beta/models/...:generateContent`，最终触发 `Invalid URL` 404；同时错误分类里使用 `includes('rate')`，会把 `generateContent` 误判为“请求频率超限”。
- 根因五：上游网络异常（`TypeError: fetch failed sending request`）时，请求可能长时间阻塞，最终才以网络错误失败，用户感知为“卡住后失败”。
- 根因六：媒体代理路由 `/m/[publicId]` 在本地存储场景拉取 `/api/files/...` 时，若使用默认 `localhost:3000` 作为上游地址，在非 3000 端口部署会触发 `ECONNREFUSED`。
- 影响：用户看到的是“参数错误”而非“模型未配置”，排障方向被误导。

## 修复内容
- 文件：
  - `src/app/api/novel-promotion/[projectId]/generate-image/route.ts`
  - `src/app/api/novel-promotion/[projectId]/regenerate-group/route.ts`
  - `src/app/api/novel-promotion/[projectId]/regenerate-single-image/route.ts`
  - `src/app/api/user/api-config/route.ts`
  - `src/lib/api-config.ts`
  - `src/lib/config-service.ts`
  - `src/lib/provider-base-url.ts`
  - `src/lib/generators/image/gemini-compatible.ts`
  - `src/app/m/[publicId]/route.ts`
- 调整点：
  - 在读取项目配置后，使用 `checkRequiredModels` 校验当前请求对应的必需模型：
    - `type=character` 需要 `characterModel`
    - `type=location` 需要 `locationModel`
  - 缺失时直接抛出 `ApiError('MISSING_CONFIG')`，消息使用 `getMissingConfigError(...)`，例如：
    - `请先在项目设置中配置"角色图像模型"`
    - `请先在项目设置中配置"场景图像模型"`
  - 图片能力配置解析失败时，也统一抛出 `MISSING_CONFIG`（不再映射为 `INVALID_PARAMS`）。
  - 避免 `details.code` 覆盖响应顶层统一错误码，改为使用 `details.reason`。
  - `getProjectModelConfig` 对 `characterModel/locationModel/storyboardModel/editModel/videoModel` 增加用户偏好回退，落实“项目 > 用户偏好 > null”。
  - 对 capability 必填/非法值错误进行用户化消息映射：
    - `CAPABILITY_REQUIRED` → `请先在项目设置中为所选图像模型配置"分辨率"`（用户配置场景对应“用户配置”）
    - `CAPABILITY_VALUE_NOT_ALLOWED` → 明确提示对应能力参数无效并要求重新选择
  - 新增 `gemini-compatible` baseUrl 校验：
    - 明确禁止把 baseUrl 配成 endpoint（如 `/images/generations`、`/v1beta/models`）
    - 在保存用户配置（PUT `/api/user/api-config`）时前置拦截，返回 `INVALID_PARAMS`
    - 在运行时读取 provider 配置时同样校验，避免错误配置进入生成链路
  - 修复 Gemini 兼容错误分类：
    - 限流判断不再使用 `includes('rate')`（避免误命中 `generateContent`）
    - `Invalid URL` 场景优先映射为“Base URL 配置错误”，给出可操作指引
  - 新增 Gemini 兼容请求超时保护：
    - 对 `generateContent` 增加显式超时（默认 120 秒，可通过 `GEMINI_COMPATIBLE_REQUEST_TIMEOUT_MS` 调整）
    - 超时统一报错为“Gemini 兼容服务请求超时”，避免 10 分钟级别无反馈
    - `fetch failed sending request` 稳定映射为网络异常，避免误导为业务参数问题
  - 修复媒体代理 `/m/[publicId]` 连通性：
    - 当签名地址是相对路径（`/api/files/...`）时，改为基于当前请求 origin 组装上游 URL，而非默认 `localhost:3000`
    - 上游 `fetch` 抛异常时显式返回 `502`（`Failed to fetch media`），避免未捕获异常导致的 500 抛栈

## 使用与行为说明
- 变更后，当项目未配置对应图片模型时：
  - 接口状态码仍为 `400`
  - 统一错误码为 `MISSING_CONFIG`
  - 文案明确指向“请先在项目设置中配置对应模型”
- 该行为遵循“显式失败、零隐式回退”原则，不做自动降级或兜底模型替换。
