# 持续上游 PR 筛选与落地流程（Playbook）

## 1. 目标与适用范围

用于团队持续吸收上游 PR 的有效修复/优化，并在 fork 内部以可控方式复现落地。

- 目标：避免重复踩坑、提升修复复用率、形成可审计知识沉淀。
- 适用：所有来自上游仓库的 PR（含 open / merged / closed）。
- 原则：**不直接照搬外部 PR**，以“结论采纳 + 本仓实现 + 本仓验证”为准。

## 2. 分级标准（优先级）

### P0（必须优先）
- 计费、数据一致性、任务状态正确性、鉴权/安全边界、生产可用性。
- 典型信号：错误归类导致不可恢复失败、任务扣费异常、核心链路超时/中断。

### P1（次优先）
- 稳定性、一致性、可观测性、关键路径体验提升。
- 典型信号：错误标准化缺口、模型/配置回退不一致、请求头/超时策略缺失。

### P2（排队）
- 低风险优化、风格一致性、非关键链路改进。

## 3. 标准作业流程

### Step A：收集
1. 拉取上游最近一轮 PR 列表。
2. 过滤与本仓架构无关项。
3. 形成候选清单（标题、链接、影响模块、初判优先级）。

### Step B：初筛
按以下问题快速打标签：
- 是否修复真实故障或高频问题？
- 是否与本仓现有模块直接对应？
- 是否需要 schema/协议变更？
- 是否可能影响计费、任务状态、重试语义？

输出：P0/P1/P2 分级结果。

### Step C：技术评审
对 P0/P1 逐项确认：
- 影响文件与调用链；
- 与本仓实现差异（是否可直接借鉴思路）；
- 回滚复杂度；
- 测试可覆盖性。

输出：
- 采纳 / 暂缓 / 拒绝
- 原因与证据路径

### Step D：落地实现
1. 在本仓按既有模式实现（不引入无关重构）。
2. 补齐最小必要测试（单测优先，必要时补集成）。
3. 本地验证通过后进入评审。

### Step E：归档沉淀
每轮必须新增或更新两类文档：
1. **采纳日志**（本轮“改了什么、为什么、证据是什么”）
   - 位置：`docs/pr-triage/P0-P1-adoption-log.md`（或按日期拆分）
2. **持续流程文档**（本 Playbook）
   - 位置：`docs/pr-triage/continuous-triage-playbook.md`

## 4. 验证清单（合并前）

- [ ] 改动范围仅覆盖目标问题，无额外功能扩展
- [ ] 每个采纳点都有对应测试或可复现实证
- [ ] 错误码/重试语义与现有规范一致
- [ ] 关键日志与可观测字段未被破坏
- [ ] 已明确回滚方案（按模块可回退）
- [ ] 文档已更新（采纳日志 + 本流程文档）

## 5. 记录模板（可直接复制）

```md
## [PR候选] <标题或主题>
- 来源：<上游PR链接>
- 分级：P0 / P1 / P2
- 结论：采纳 / 暂缓 / 拒绝
- 影响模块：<file paths>
- 核心风险：<1-3条>
- 落地改动：<本仓改动点>
- 验证证据：<测试文件或命令结果>
- 回滚点：<可逆改动说明>
```

## 6. 角色分工（多人协作）

- Triage Owner：维护候选池、主持分级与结论。
- Implement Owner：执行代码落地与测试补齐。
- Reviewer：验证结论与风险边界，确认文档完整性。

同一人可兼任多个角色，但每轮至少保证“实现者与评审者分离”。

## 7. 本仓执行约束

1. 保持小步提交：每个采纳点可独立回滚。
2. 不做“顺手重构”：避免扩大变更面。
3. 仅在必要处新增文件，优先复用现有结构。
4. 所有结论必须落文档，避免只存在聊天记录。
